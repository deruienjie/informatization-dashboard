工单结案与在制材料处理

已整理
业务归属：成本核算 → 工单成本结算
触发场景：生产工单发料后，实际产量少于计划，剩余材料在车间（在制），手工结案时被强算成损耗
核心问题：发料75颗 → 用66颗 → 结案 → 9颗在制被算成损耗 → 成本虚高
IT要做什么：
① 确认ERP是否支持"有在制不允许手工结案"
② 确认"生产领料强制按BOM"的系统设置
③ 确认"其他出库"流程是否跑通
④ 如有必要，提系统变更需求
业务流程
① 发料 按BOM发75颗 → ② 生产 实际用66颗 → ③ 入库 部分完工入库 → ④ 手工结案 ❌ → 9颗在制变损耗
系统机制：手工结案把所有"已领未用"的材料强制计入完工成本，不管实物是否还在车间
解决方案（财务群已达成一致）
✅ 损耗走"其他出库"
正常的损耗不通过生产领料走，单独用"其他出库"单据，便于追踪和管理
✅ 生产领料强制按BOM
不能超BOM领料，从系统层面做控制
✅ 有在制不允许手工结案（核心控制点）
工单上还有材料没退（在制数量 ≠ 0），系统禁止做手工结案。必须先退料或在制归零，才能结案。
💡 学习要点：这个案例的核心是"手工结案会把在制材料强算成损耗"——这是ERP成本核算的一个常见坑。以后财务说"成本不对"，你可以先问一句："是不是手工结案的时候还有在制没退？"

应付单下推增值税发票金额不一致

已整理
业务归属：应收应付管理 → 应付单下推增值税发票
触发场景：应付单审核保存后，下推生成增值税发票时，金额字段（未税金额/税额/价税合计）出现不一致，财务需手动调整
核心问题：下推时单据转换规则用"公式重算"而非"取上游值"——发票从原始单价×数量重新计算金额，财务在应付单上的手工调整全部丢失
IT要做什么：
① 确认应付单下推发票的单据转换规则是否配置了"公式计算"
② 将3个金额字段改为"取上游单据值"（不重算）
③ 测试验证：开AP单→下推发票，3个金额字段是否完全一致
④ 回访财务确认问题解决
业务流程
① 入库 采购收货入库 → ② AP单 生成应付单，财务核对供应商发票后可能手工调金额 → ③ 保存 应付单审核保存（3个金额已确认） → ④ 下推 ❌ → 发票系统不复制确认值，而是从单价×数量重算
系统机制：发票系统拿到原始"单价×数量"重新跑计算，不走应付单已经舍入/手工调整过的最终金额
根因分析
🔴 根因1：映射方向错
配置的是"公式重算"（发票金额从零算），而不是"取上游值"（原样复制应付单金额）
🔴 根因2：舍入偏差
金蝶内部用6位精度计算，但上一轮算税额时已在某一步做了2位舍入。发票从6位精度重新跑，舍入方向可能不同
🔴 根因3：手动调整丢失（最常见）
供应商发票与系统算出的值差0.01（双方舍入规则不同），财务手工改AP单金额对齐供应商发票。但下推发票时，发票系统拿"原始单价×数量"重新算，完全无视财务改过的值。
解决方案
✅ 核心改法：改单据转换规则
路径：开发平台 → 业务流 → 应付单下推增值税发票 → 单据转换规则
将下面3个金额字段的映射从"公式"改为"取上游单据值"：
· 未税金额 ← 应付单.未税金额
· 税额 ← 应付单.税额
· 价税合计 ← 应付单.价税合计
注意：只改金额字段，不改"单价"和"数量"
✅ 治本逻辑
应付单存的3个金额 = 财务最终确认的值（含任何手工调整）
发票的3个金额 = 直接复制应付单的值（一分不差）

区别：
取上游值 = 复制财务确认的最终结果
公式重算 = 从原始数据重新跑，忽略人工调整
✅ 额外排查清单（改完映射后再验）
① 金额精度：确认单价/金额/税额精度≥2位（路径：系统建模→应付管理→参数设置）
② 单据参数：确认"税额计算采用四舍五入"规则统一
③ 外币单据：若是外币AP单，确认汇率已锁（有锁图标=已锁）
④ 部分开票：若是分批开票而非全额下推，金额不一致是正常业务逻辑
🔬 技术深挖：为什么"从零重算"会出错？（展开）

核心矛盾：应付单存的3个金额 ≠ 从"单价×数量"直接算出的3个金额，因为应付单保存前经历了至少一轮舍入。


举例：财务在AP单上手动把税额从751.71改成751.70（对齐供应商发票）。AP单保存的税额= 751.70。

发票系统说："你和供应商发票对齐了OK，但我要从原始数据重算：单价×数量÷1.13×0.13 = 751.71。所以发票上显示 751.71，和你的AP单差0.01。"


本质：配置选"公式计算" = 系统假定了"没人改过原始数据"。但实际业务中，财务经常需要手工对齐供应商发票——这些人工调整在"公式计算"路径下全部丢失。

💡 学习要点：这个案例的核心是理解 ERP 的"单据转换规则"决定了下推数据的准确性。默认模板为了灵活性做了公式重算，但对国内采购全额开票+财务手动核对场景，应该改为"取上游值"。以后财务说"下推后金额变了"，第一反应就是：确认单据转换规则的3个金额字段是不是配了公式重算。

付款申请单下推付款单显示超额

已整理
业务归属：应收应付管理 → 付款申请 → 付款单
触发场景：退款申请单下推付款单时提示"超额、无法下推"，十几笔6月29日的单据都遇到
核心问题：两套付款入口共享同一个应付额度——旧流程（应付单→付款单）已占了一部分，新流程（应付单→付款申请单→付款单）再推，系统认为重复/超额
IT要做什么：
① 查旧流程遗留的付款单（未审核的那些）→ 作废/删除
② 确认所有付款统一从"付款申请单"下推
③ 系统配置禁用"应付单→付款单"直接入口
④ 过渡期内帮助财务逐单清理
时间线
6月30日前 旧流程：应付单 → 付款单（直接付了一部分）
6月30日后 新流程：应付单 → 付款申请单 → 付款单（统一口径）
现在 旧付款单占着额度没释放 → 新付款申请单被挡 → 超额 ❌
系统控制：同一笔应付单下推的所有付款金额之和不能超过应付金额本身。不管从哪个入口付的，全部累加。
根因
🔴 流程口径变更
6月30日起统一要求走 应付单→付款申请单→付款单。但旧流程应付单→付款单的历史数据还在——那些已下推（哪怕未审核）的付款单占用着应付额度。
🔴 金蝶额度控制
应付单有一个"可付金额 = 应付金额 - 已付金额"。旧付款单计入"已付"，新付款申请单计入"要付"。二者累加 ≥ 应付额 → 超额拦截。
🔴 分批支付的特殊性
采购经常不是整单付款，是分批付的。但分批付的所有批次必须走同一个入口——不能第一批走应付单直接推付款单，第二批走付款申请单。混着走系统无法归口统一核算。
解决方案
✅ 短期（过渡清理）
① 查出这十几笔单据关联的旧流程付款单
② 未审核的付款单 → 作废/删除
③ 已审核的 → 反审核后冲销，从付款申请单重新推
④ 确认所有付款批次统一走"付款申请单"
✅ 长期（系统配置）
路径：开发平台 → 业务流 → 应付单下推付款单
禁用"应付单→付款单"入口
只保留：应付单 → 付款申请单 → 付款单

效果：以后不可能再从两个路径混着走，杜绝同类问题。
💡 学习要点：这个案例的核心是"流程口径变更导致的历史数据冲突"——不是系统Bug，是两套路径共享同一个额度池。以后遇到"下推超额"，第一反应就是查：这个上游单是不是被另外一个入口推过了？ 尤其是流程改造过渡期，这种问题会集中爆发。
